
## Файлы и их роли

### `main.py` — Основное приложение (18KB)

**Содержит:**
- Класс `OzonSupplyBot` — главный класс бота
- `SupplyForm` и `SetupForm` — FSM состояния (пошаговый диалог)
- Обработчики команд (`/start`, `/setup`, `/create`)
- Обработчики колбеков (нажатие кнопок)
- Методы показа интерфейса (кластеры, даты, подтверждение)

**Поток взаимодействия:**
1. Пользователь пишет `/start` → `cmd_start()`
2. Бот проверяет, есть ли учетные данные → если нет, предлагает `/setup`
3. Пользователь выполняет `/setup` → переход в `SetupForm.waiting_for_client_id`
4. Ввод Client-ID → `SetupForm.waiting_for_api_key`
5. Ввод API-Key → сохранение в БД, очистка состояния
6. Пользователь выполняет `/create` → начало `SupplyForm`
7. Пошаговый диалог: SKU → количество → тип доставки → кластеры → дата → подтверждение

### `tgbot_db.py` — Работа с БД (2.8KB)

**Таблица `tg_users`:**
```sql
CREATE TABLE tg_users (
    telegram_id   INTEGER PRIMARY KEY,  -- Telegram User ID
    username      TEXT,                 -- @username
    client_id     TEXT NOT NULL,        -- Ozon Client-ID
    api_key       TEXT NOT NULL,        -- Ozon API-Key
    created_at    TEXT,
    updated_at    TEXT
)
```

**Функции:**
- `init_users_table()` — создание таблицы
- `save_user_credentials()` — сохранение/обновление учетных данных
- `get_user_credentials()` — получение учетных данных
- `user_exists()` — проверка регистрации

### `supply_flow.py` — Полный поток создания поставки (9.7KB)

**Шаги:**
1. `get_product_by_sku(sku)` — поиск товара в БД по SKU
2. Проверка остатков товара
3. Создание черновика поставки
4. Скоринг — проверка доступности на кластерах
5. Получение таймслотов
6. Создание финальной поставки
7. Возврат результата

**Функция:**
```python
async def create_supply_full_flow(
    client: OzonClient,
    sku: str,
    quantity: int,
    delivery_type: str,
    cluster_ids: list[int],
    target_date: str,
) -> dict
```

## FSM (Finite State Machine) — Состояния диалога

### Состояния для `/setup`:

```
START
  ↓
SetupForm.waiting_for_client_id (ввод Client-ID)
  ↓
SetupForm.waiting_for_api_key (ввод API-Key)
  ↓
[Сохранение в БД]
  ↓
КОНЕЦ
```

### Состояния для `/create`:

```
START
  ↓
SupplyForm.waiting_for_sku (введи SKU)
  ↓
SupplyForm.waiting_for_quantity (сколько штук)
  ↓
SupplyForm.waiting_for_delivery_type (выбери кнопку)
  ↓
SupplyForm.waiting_for_clusters (выбери кластеры)
  │
  └─→ [Пользователь нажимает на кластеры галочками]
  │
  └─→ [Нажимает "Далее"]
  ↓
SupplyForm.waiting_for_date (выбери дату)
  ↓
SupplyForm.confirming_supply (проверь и подтверди)
  ↓
[Вызов supply_flow.create_supply_full_flow()]
  ↓
КОНЕЦ
```

## Интеграция с основным проектом

### Используемые компоненты из `..` (родителя):

1. **`ozon_client.py`** — API клиент Озона
   - `OzonClient` класс с методами для работы с Озон API
   - `OzonAPIError` исключение

2. **`db.py`** — основная база данных
   - Таблица `products` — товары
   - Таблица `stocks` — остатки
   - Таблица `warehouses` — склады
   - И другие...

### Расширение БД:

Бот добавляет новую таблицу `tg_users` в существующую БД `warehouse.db`:
- Не модифицирует существующие таблицы
- Использует ту же БД файл
- Изолированные данные пользователей

## Поток создания поставки (детально)

### 1. Ввод данных (в Telegram)

```
Пользователь: /create
Бот: Введи SKU
Пользователь: вак1
Бот: Сколько штук?
Пользователь: 100
Бот: [Кнопки: Прямая поставка | Кросс-докинг]
Пользователь: [Прямая поставка]
Бот: [Кластеры с галочками]
Пользователь: [Выбирает кластеры, нажимает "Далее"]
Бот: [Даты для поставки]
Пользователь: [Выбирает дату]
Бот: [Подтверждение]
Пользователь: [Подтверждает]
```

### 2. Обработка в `supply_flow.create_supply_full_flow()`

```
1. Получить товар из БД по SKU
   ├─ SELECT product_id, fbo_sku FROM products WHERE offer_id = ?
   └─ Возвращает: {product_id: 123, fbo_sku: 456789, name: "вак"}

2. Проверить остатки
   ├─ SELECT present, reserved_ozon, reserved_local FROM stocks
   ├─ Вычислить: free = present - reserved_ozon - reserved_local
   └─ Если free < quantity → ОШИБКА

3. Создать черновик поставки
   ├─ client.create_draft(items=[{sku: 456789, quantity: 100}])
   └─ Возвращает: {draft_id: 999, cluster_id: 1, cluster_name: "Москва"}

4. Проверить доступность на кластерах (скоринг)
   ├─ client.get_draft_info(draft_id=999)
   ├─ Анализирует: какие кластеры доступны для данного товара
   └─ Возвращает: clusters[] с availability_status для каждого склада

5. Получить таймслоты (доступные окна приемки)
   ├─ client.get_timeslots_v2(draft_id=999, cluster_id=..., ...)
   └─ Возвращает: [{date: "2024-04-01", timeslots: [{from: "08:00", to: "12:00"}]}]

6. Создать финальную поставку
   ├─ client.create_supply_v2(draft_id=999, timeslot_from="...", timeslot_to="...")
   └─ Возвращает: {draft_id: 999, error_reasons: []}

7. Получить статус создания (опционально)
   ├─ client.get_supply_create_status(draft_id=999)
   └─ Возвращает: order_id если успешно
```

### 3. Ответ пользователю

```
Бот показывает результат:
✅ Поставка создана!
Order ID: 123456789
Кластер: Москва (ЦОД-3)
Дата: 2024-04-01 08:00-12:00
```

## Безопасность

### ✅ Что защищено:

1. **Client-ID и API-Key**
   - Хранятся в локальной SQLite БД
   - Не отправляются никуда (кроме Озон API)
   - Не видны в логах

2. **Telegram Bot Token**
   - Хранится в `.env` файле
   - Не коммитится в git (в `.gitignore`)

3. **Изоляция пользователей**
   - Каждый пользователь имеет свои учетные данные
   - Один User ID = одна запись в tg_users

### ⚠️ Что нужно помнить:

- Компьютер, где запущен бот, должен быть в безопасности
- БД `warehouse.db` содержит чувствительные данные
- При необходимости можно зашифровать API ключи (но в текущей версии просто хранятся)

## Расширяемость

Архитектура позволяет легко добавлять:

1. **Новые команды** — добавить в `main.py`
2. **Новые сценарии поставок** — расширить `supply_flow.py`
3. **Отслеживание поставок** — добавить новую таблицу в БД
4. **Интеграции** — использовать `tgbot_db` для доступа к учетным данным
